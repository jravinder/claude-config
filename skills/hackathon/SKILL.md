# Hackathon Workflow

How to build when the clock is ticking and the window is 4-48 hours.

## Before the Clock Starts

- Research the problem space (Agent 2)
- Create repo with config, proxy, and deploy scripts
- Fill out submission form with placeholders
- Read the judging criteria. Build for the judges, not for yourself.

## Hour 0-1: Lock the Idea

- State the problem in one sentence
- Identify the user (one specific person, not "everyone")
- Define the demo moment (the 10-second wow)
- Choose the stack (use what you know, not what's new)

## Hour 1-4: Build

- Builder agent ships features
- Content agent writes README, website copy, submission text in parallel
- Don't polish. Ship ugly, then refine.
- Commit every 15 minutes. If you lose power, you lose nothing.

## Hour 4-5: Demo Prep

- Screenshots of every screen
- Demo script: what you show, what you say, in what order
- Test the live URL from a different device
- Record the video BEFORE the last 30 minutes

## Last 30 Minutes

- Final submission form review
- One last deploy and smoke test
- Do not add features. Only fix what's broken.

## What Wins

- **Story > code.** Judges remember the problem you solved, not the architecture.
- **Working > complete.** A demo that runs beats a pitch deck about what it could do.
- **Specific > broad.** "Helps volunteer baseball coaches in Austin" beats "AI for sports."
- **Honest > polished.** Say what it doesn't do yet. Judges respect that.

## What Loses

- Features nobody asked for
- Broken demos (test from a clean browser)
- Claiming something the code doesn't actually do
- Spending the last hour on CSS instead of the submission form
